## Reglas que costaron tiempo hoy (no re-aprenderlas)

- **El CLI de Supabase ordena los archivos por NOMBRE y el ledger por VERSIÓN.** Si los dos órdenes no
  coinciden, el merge-join se desalinea y `db push` se bloquea **aunque la base esté perfecta**. Fue el
  caso de `009` vs `0095` (`'5'`=53 < `'_'`=95). Por eso el rename a `0090`. Persiste en CLI 2.109.1.
- **Nunca `repair --status reverted` sobre algo aplicado.** El CLI lo sugiere; le mentiría al ledger.
- **`db query --linked` cuelga** → Management API (`read_only:true` + smoke `25006`).
  En macOS **no hay `timeout`**: usar `perl -e 'alarm 90; exec @ARGV' …`.
- **`brew upgrade supabase` rompe `node@22`** (arrastra `simdutf`) → `dyld: Library not loaded`.
  Se arregla con **`brew reinstall node@22`**, y después re-verificar tests + build.
- **Falso 200 al verificar un deploy:** el fallback SPA responde `index.html` (~2,7 kB) con 200 a
  cualquier `/assets/*.js` inexistente, y `/index.html` redirige 308 (un `curl` sin `-L` trae 0 bytes).
  Verificar **caminando el grafo de chunks** → [`scripts/verificar-deploy-cloudflare.sh`](scripts/verificar-deploy-cloudflare.sh).
- **`supabase/.temp/linked-project.json` SÍ existe** (docs viejos decían que no) y trae el `name`
  (`satori-staging` vs `satori-app`) — el desempate cuando el ref solo no alcanza.
