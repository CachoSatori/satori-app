## 🔴 Hallazgo que bloqueaba todo: **el asiento de apertura NO existía**

`REPORTE-PASE-STAGING.md` §3 y `TRIAGE-TARJETA.md` dan por sembrado el asiento de apertura
(₡744.575 / $3.441, fechado **2026-07-23**). **En la base había CERO filas** con
`subcategory = 'Apertura pozo'` — verificado por SQL directo, no por un filtro en JS:

```sql
select * from public.cash_movements where subcategory = 'Apertura pozo';  -- []
```

### Por qué era bloqueante

La tarjeta y el "debería" **no cuentan desde el corte: cuentan desde el asiento de apertura**
(`fechaAperturaPozo` → `base = movimientos con fechaOperativa >= apertura`). Con la apertura
ausente, `fechaAperturaPozo` devuelve `null` y `tarjetaPozo` cae a `base = movements` — **toda
la historia**, 1.430 filas desde enero. Es exactamente el escenario contra el que advierte el
comentario de `cierrePozo.ts`: *el histórico no tiene ancla y el pozo acumulado da negativo*.

Mover el corte **sin sembrar la apertura** le habría puesto un número basura en la pantalla.

### Qué se hizo

Se sembró el asiento con **fecha 2026-07-22** (no 23), que es lo coherente con el corte nuevo:

```bash
node --import ./scripts/t0-reconciliacion-cajas/register.mjs \
  scripts/refresh-staging/seed-apertura.ts --fecha 2026-07-22
```

**La cifra no se inventó ni se copió del reporte viejo** — se verificó contra el conteo físico
sellado del cierre completo del 21/07, leyendo la base:

### La env var gana sobre el fallback

`resolverCorte()` devuelve `VITE_POZO_CORTE` si es un `YYYY-MM-DD` real, y solo cae al fallback
si viene vacía o mal formada (con aviso en consola). Entonces:

- **Si `VITE_POZO_CORTE` NO está cargada en Cloudflare** → manda este commit. Nada que hacer.
- **Si SÍ está cargada** → gana ella y este commit queda inerte. Para que las dos vías digan lo
  mismo, **debe valer exactamente `2026-07-22`**. Si hoy tuviera `2026-07-23`, *ese* sería el
  corte vivo y la pantalla no se movería — hay que corregirla en el dashboard.

No se puede leer ni setear esa variable desde el repo (config externa de Cloudflare Pages). El
bundle desplegado dice `2026-07-22`, lo cual es consistente con que la variable no esté cargada
**o** con que valga ya `2026-07-22`; las dos son correctas.
